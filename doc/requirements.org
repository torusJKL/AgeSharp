** 1. Project Overview
:PROPERTIES:
:CUSTOM_ID: project-overview
:END:

- Project Name: AgeSharp
- Type: C# .NET 10 Library + CLI Application
- Core Functionality: Implementation of the Age file encryption format (age-encryption.org/v1) in C#, with a NuGet-compatible library and CLI tool interoperable with the reference implementation.

** 2. Goals
:PROPERTIES:
:CUSTOM_ID: goals
:END:

1. Create a C# .NET 10 library that implements the Age encryption specification (v1)
2. Publish the library as a NuGet package for consumption by other projects
3. Build a CLI application that provides feature parity with the reference implementation (filippo.io/age)
4. Ensure unit tests using xUnit

**** 4.2.1 Core Cryptographic Parameters

| Parameter              | Value                              |
|-----------------------|------------------------------------|
| File key size         | 16 bytes (128-bit)                |
| Chunk size            | 64 KiB (65536 bytes)              |
| Chunk counter         | 11 bytes (big-endian)             |
| Final chunk marker    | 0x01 (last), 0x00 (intermediate) |
| Header MAC            | HMAC-SHA-256                      |
| MAC key salt          | empty                              |
| MAC key info          | "header"                           |
| Payload nonce         | 16 bytes (CSPRNG, unique per file)|
| Payload key salt      | nonce                              |
| Payload key info      | "payload"                          |

***** Base64 Encoding Requirements

- Must use standard Base64 (RFC 4648, Section 4)
- No padding characters (=) allowed
- Must generate canonical Base64 (RFC 4648, Section 3.5)
- Must reject non-canonical encodings
- Must reject encodings with padding characters

**** 4.2.2 Key Components

| Component    | Description                                           | Public API |
|--------------+-------------------------------------------------------+------------|
| Age          | Main entry point for encryption/decryption            | Yes        |
| AgeInspector | File inspection for age-inspect command               | Yes        |
| AgeHeader    | Parses and generates the textual header               | Internal   |
| AgePayload   | Handles binary payload encryption/decryption          | Internal   |
| Recipients   | Interface and implementations for all recipient types | Internal   |
| Identities   | Interface and implementations for all identity types  | Internal   |
| Keys         | Key generation, derivation, and encoding utilities    | Internal   |

**** 4.2.2 Supported Recipient Types

| Type               | Identity Format         | Recipient Format | Description                  | Status |
|--------------------+-------------------------+------------------+------------------------------+--------|
| X25519             | AGE-SECRET-KEY-1...     | age1...          | Standard X25519 key exchange | Done   |
| Scrypt             | (passphrase string)     | (passphrase)     | Password-based encryption    | Done   |
| ML-KEM768-X25519   | AGE-SECRET-KEY-PQ-1...  | age1pq1...       | Hybrid post-quantum (X-Wing) | Future |
| P256-Tag           | (plugin-based)          | age1tag1...      | P-256 ECDH with tag          | Future |
| ML-KEM768-P256-Tag | (plugin-based)          | age1tagpq1...    | Hybrid post-quantum tagged   | Future |

**** 4.2.2.1 Key File Formats

***** X25519 Keys

- *Identity (private key)*: Bech32 encoding with HRP =AGE-SECRET-KEY-=
  #+BEGIN_EXAMPLE
  AGE-SECRET-KEY-1GFPYYSJZGFPYYSJZGFPYYSJZGFPYYSJZGFPYYSJZGFPYYSJZGFPQ4EGAEX
  #+END_EXAMPLE
- *Recipient (public key)*: Bech32 encoding with HRP =age=
  #+BEGIN_EXAMPLE
  age1zvkyg2lqzraa2lnjvqej32nkuu0ues2s82hzrye869xeexvn73equnujwj
  #+END_EXAMPLE
- Identity is 32 bytes, encoded with Bech32
- Recipient is derived via X25519 scalar multiplication

***** Scrypt (Passphrase)

- Identity: Passphrase string (any Unicode text)
- Recipient: Passphrase string (same as identity for encryption)
- No encoding needed - passphrase is used directly as input to scrypt

***** Identity File Format

The identity file (used by CLI) contains one or more identities:

#+BEGIN_EXAMPLE
# age identity file
AGE-SECRET-KEY-1GFPYYSJZGFPYYSJZGFPYYSJZGFPYYSJZGFPYYSJZGFPYYSJZGFPQ4EGAEX
#+END_EXAMPLE

- Lines starting with =#= are comments
- Each non-comment line contains one identity (private key or passphrase)
- Multiple identities can be listed for decryption with multiple keys

**** 4.2.3 Cryptographic Primitives

| Primitive         | Implementation                           |
|-------------------+------------------------------------------|
| ChaCha20-Poly1305 | System.Security.Cryptography             |
| X25519            | System.Security.Cryptography             |
| HKDF-SHA-256      | System.Security.Cryptography             |
| HMAC-SHA-256      | System.Security.Cryptography             |
| Scrypt            | Norgerman.Cryptography.Scrypt (NuGet)    |
| Bech32            | Nano.Bech32 (NuGet)                      |
| Base64            | System.Convert with canonical validation |

*Note:* ML-KEM-768 will be needed for future post-quantum support.

**** 4.2.3.1 Scrypt Parameters

The scrypt key derivation follows RFC 7914 with the following parameters:

| Parameter        | Value                                      |
|------------------|-------------------------------------------|
| Salt size        | 16 bytes (randomly generated per file)   |
| Work factor (N)  | 2^logN (logN provided in stanza)         |
| Block size (r)   | 8                                         |
| Parallelization  | 1                                         |
| Output length    | 32 bytes                                  |
| Salt prefix      | "age-encryption.org/v1/scrypt"           |
| Nonce            | 12 bytes of 0x00                          |

***** Default Work Factors

| Context        | Default logN | Max logN (configurable) |
|----------------|--------------|------------------------|
| Encryption     | 18           | -                      |
| Decryption     | -            | 22                     |

***** Stanza Format

#+BEGIN_EXAMPLE
-> scrypt <base64-salt> <logN>
<body>
#+END_EXAMPLE

- First argument: "scrypt"
- Second argument: base64-encoded 16-byte salt
- Third argument: base-two logarithm of work factor (decimal, no leading zeros)
- Body: ChaCha20-Poly1305 encrypted file key (32 bytes)

***** Validation Rules

- Salt MUST be exactly 16 bytes when decoded
- Work factor MUST match regex: ^[1-9][0-9]*$ (no leading zeros)
- Body MUST be exactly 32 bytes before decryption attempt
- Scrypt stanza MUST be the only stanza in header (no mixing with other recipient types)

**** 4.3.2 CLI Usage

#+BEGIN_SRC bash
# Encrypt
age -o output.age -r RECIPIENT... [-R RECIPIENTS_FILE] [--armor] [--passphrase] [INPUT]

# Decrypt
age -d [-i IDENTITY_FILE]... [-o OUTPUT] [INPUT]

# Key generation
age-keygen [-o key.txt]

# Inspection
age-inspect [--json] FILE
#+END_SRC

*Note:* =--pq= flag for post-quantum keys is a future feature.

**** 4.3.3 CLI Options

| Flag                  | Description                       |
|-----------------------+-----------------------------------|
| -e, --encrypt         | Encrypt (default)                 |
| -d, --decrypt         | Decrypt                           |
| -o, --output          | Output file path                  |
| -a, --armor           | Use ASCII armor (PEM)             |
| -p, --passphrase      | Encrypt with passphrase           |
| -r, --recipient       | Encrypt to recipient (can repeat) |
| -R, --recipients-file | Read recipients from file         |

*** 7.1 Passphrase Encoding Requirements

The implementation MUST support Unicode passphrases following modern best practices:

| Requirement        | Specification                                          |
|--------------------+--------------------------------------------------------|
| Encoding           | UTF-8 for cryptographic operations                     |
| Normalization      | Unicode NFC (Normalization Form Canonical Composition) |
| Length calculation | Each Unicode code point counts as one character        |
| Maximum length     | At least 64 characters                                 |
| Minimum length     | 1 character (empty passphrases rejected)               |

**** Implementation Notes

- Passphrase strings should be normalized to NFC before processing with scrypt
- Use ~System.Text.Encoding.UTF8.GetBytes~ to convert normalized passphrase to bytes
- Each Unicode code point should be counted as a single character for length validation
- Reject overlong UTF-8 sequences and invalid Unicode encodings
- This aligns with NIST SP 800-63B-4 (2024) password guidelines
